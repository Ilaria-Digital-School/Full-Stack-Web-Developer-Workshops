# Workshop: adding a .gitignore file adapted to an HTML/CSS project

**Theoretical Recap** A .gitignore file tells Git which files and directories to intentionally leave untracked. When Git scans your project folder, it reads .gitignore and skips anything matching the listed patterns — those files never appear in git status, are never staged, and are never committed. This is critical because some files must never enter a repository: files containing secrets or API keys, files auto-generated by your operating system (like .DS_Store on macOS or Thumbs.db on Windows), editor configuration folders (like .vscode/ or .idea/), and dependency folders that can be reinstalled at any time (like node_modules/ in future JavaScript projects). The mental model to keep in mind: your repository should contain only the source code a teammate needs to clone, run, and understand your project. Everything else is noise — or worse, a security risk. A common beginner mistake is creating the .gitignore file after already having committed unwanted files. In that case, Git continues tracking those files even if you add them to .gitignore afterward, because they are already in the history. The correct reflex is to create and configure .gitignore before the first commit, or at minimum before committing any sensitive or auto-generated files. Another frequent mistake is placing the .gitignore file in the wrong directory — it must be at the root of the repository to apply globally to the project.
