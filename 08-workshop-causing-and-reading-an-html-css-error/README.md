# Workshop: causing and reading an HTML/CSS error

**Theoretical Recap** When you write HTML or CSS, errors are inevitable — and learning to read them is one of the most critical skills you will develop as a developer. An HTML error typically occurs when the browser encounters markup it cannot properly interpret: a missing closing tag, a misspelled tag name, an improperly nested element, or a missing required attribute. CSS errors happen when a property name or value is invalid, a selector is malformed, or a declaration block is not properly closed. Unlike compiled languages, browsers are generally forgiving with HTML — they will often render something even when the code is broken. This is a trap: the page may look fine visually while the underlying structure is corrupted. CSS errors are typically silent too: an invalid rule is simply ignored, with no visible warning in the browser window. This is exactly why DevTools exists. The browser's console and Elements panel are your primary diagnostic tools. In the console, JavaScript errors will appear in red, and some HTML/CSS issues will generate warnings. In the Elements panel, you can inspect the rendered DOM tree and compare it to what you actually wrote — discrepancies reveal how the browser interpreted (or misinterpreted) your code. The Styles panel shows which CSS rules are active, overridden (shown with a strikethrough), or invalid (shown with a warning icon or simply absent).
