# Lab: installing Git and setting up user.name/user.email

THEORETICAL RECAP Git is a distributed version control system. In plain terms, it tracks every change you make to your files over time, lets you roll back to previous states, and enables collaboration with other developers without overwriting each other's work. Think of it like a detailed logbook for your codebase: every save (called a 'commit') is a timestamped entry authored by a specific person. This is why Git needs to know who you are before you can do anything meaningful with it. When you install Git on your machine and run it for the first time, it does not know your identity. You must explicitly tell it your name and email address using two global configuration commands. These values are attached to every commit you create — they are not credentials for authentication, just metadata that identifies authorship. A very common beginner mistake is skipping this setup entirely, then discovering later that all commits are attributed to an anonymous or incorrect user, which becomes a real problem when working in a team or publishing to platforms like GitHub. Mental model: Git lives on your machine (local), and your user.name and user.email are like signing your name at the bottom of every document you submit. No signature = unprofessional, untrackable, and sometimes rejected by remote repositories. Another common mistake is confusing your Git email with your GitHub login email — they should match if you want your contributions to be recognized on GitHub.
