# Workshop: making a form both responsive and accessible

THEORETICAL RECAP A responsive form adapts its layout and sizing to any screen width, while an accessible form ensures that all users — including those relying on screen readers, keyboard navigation, or assistive technologies — can interact with it without friction. These two concerns are deeply intertwined and must be addressed simultaneously, not as an afterthought. The key mental model is this: a form is a structured dialogue between a user and a system. Every field, label, error message, and button is part of that dialogue. On a small screen, this dialogue must remain clear and comfortable (stacked layout, large enough tap targets, readable text). For accessibility, the dialogue must be unambiguous even without visual context: labels must be programmatically associated with their inputs via the 'for' and 'id' attributes, error messages must be announced via 'aria-describedby', focus states must be visible, and color contrast must meet WCAG AA minimums (4.5:1 for normal text). Common mistakes include: using placeholder text as a substitute for labels (placeholders disappear on focus and are not reliably read by screen readers), relying solely on color to indicate errors, removing the default focus outline without replacing it with a custom one, and forgetting to set 'type' attributes on inputs (which degrade mobile keyboard experience). A simple mental check before submitting any form work: can I complete this form using only a keyboard? Can I understand every field without seeing the page?
