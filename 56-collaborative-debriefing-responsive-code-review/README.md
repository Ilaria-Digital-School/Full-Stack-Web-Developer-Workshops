# Collaborative debriefing: responsive code review

**Theoretical Recap** Responsive design is the practice of building web interfaces that adapt fluidly to any screen size, device, or context. The core mental model is simple: start from the smallest viewport (mobile-first), define a solid base layout, then progressively enhance it using media queries, Flexbox, and Grid as screen real estate grows. Flexbox handles one-dimensional layouts — aligning and distributing items along a single axis — while CSS Grid excels at two-dimensional layouts, managing both rows and columns simultaneously. Media queries act as conditional gates: they apply specific CSS rules only when certain viewport conditions are met (e.g., min-width: 768px). Responsive images (srcset, sizes) ensure bandwidth efficiency by serving appropriately sized assets. Accessibility concerns — sufficient color contrast, legible font sizes, visible focus indicators — must be considered at every breakpoint, not as an afterthought. Common mistakes include using fixed pixel widths instead of relative units (%, rem, vw), forgetting to set the viewport meta tag, neglecting keyboard navigation at all breakpoints, applying desktop-first thinking that creates overrides instead of clean progressive enhancements, and ignoring image optimization. A quick mental check before any responsive implementation: Is the layout fluid? Are breakpoints content-driven rather than device-driven? Are interactive elements reachable and visible on all screen sizes? Is performance considered?
