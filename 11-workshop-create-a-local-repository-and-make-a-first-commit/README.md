# Workshop: create a local repository and make a first commit

**Theoretical Recap** Git is a distributed version control system that tracks changes in your files over time. Think of it as a detailed logbook for your project: every meaningful change is recorded, labeled, and reversible. The core mental model is simple — a Git repository is a folder that Git is actively watching. Inside that repository, your work goes through three stages: the working directory (where you edit files), the staging area (where you prepare changes to be saved), and the commit history (the permanent record of saved snapshots). A commit is a snapshot of your staged changes at a given moment. Each commit has a unique identifier (a hash), an author, a timestamp, and a message describing what changed. A good commit message is short, specific, and written in the imperative: 'Add index.html structure' rather than 'stuff done' or 'update'. Common mistakes at this stage: forgetting to run `git init` before trying to commit, staging nothing before committing (which results in an error), or writing vague commit messages that make the history unreadable. Also, be aware that `git init` creates a hidden `.git` folder — do not delete or modify it manually. Mini example: you create a folder called `my-project`, run `git init` inside it, create a file called `index.html`, run `git add index.html`, then `git commit -m "Add initial index.html file"`.
